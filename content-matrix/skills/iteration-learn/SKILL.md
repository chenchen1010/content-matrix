# 迭代学习

对比 AI 生成的原稿与用户修改后的终稿，提取跨篇的共性修改规律，写入迭代规律文件。下次 content-generator 生成内容时会自动读取并遵守这些规律。

> **输出路径规则**：所有产出写入当前项目的 Obsidian vault。
> 目录结构见 `content-matrix/schemas/vault-structure.md`。

## 何时使用

当用户说"执行迭代学习""学习我的修改""进化一下""对比初稿和终稿"时使用此 skill。

## 工作流程

### Step 1: 扫描待分析文章

扫描当前项目 `内容产出/` 中已发布且尚未做过迭代分析的文章：

```bash
VAULT_PATH=$(obsidian-cli print-default --path-only)
PROJECT="$VAULT_PATH/项目/{项目名}"
CONTENT_DIR="$PROJECT/内容产出"
DRAFT_DIR="$PROJECT/原稿存档"

# 找到所有已发布的文章
grep -r -l "status: 已发布" "$CONTENT_DIR/" --include="*.md" 2>/dev/null
```

对每篇已发布文章：
1. 读取 frontmatter 中的 `draft_path` 字段
2. 检查对应原稿是否存在
3. 检查原稿的 `iterated` 字段是否为 `false`
4. 只保留同时满足以上条件的文章

如果用户指定了平台，只扫描该平台子目录。

**判断是否足够：**
- 待分析文章 ≥ 2 篇 → 继续
- 待分析文章 < 2 篇 → 告知用户"目前只有 {N} 篇待分析，建议积累 2 篇以上再执行迭代学习，这样才能提取出有意义的共性规律"，结束

### Step 2: 逐篇对比

对每篇文章，读取原稿和终稿，按以下维度分析：

```
请对比以下两个版本，逐段分析用户做了哪些修改。

## 原稿（AI 生成）
{原稿内容}

## 终稿（用户修改后发布）
{终稿内容}

## 分析维度

1. **删除了什么**：用户删掉了哪些段落、句子或表述？这说明用户觉得什么是多余的。
2. **新增了什么**：用户加入了哪些原稿没有的内容？这说明用户觉得什么是缺失的。
3. **替换了什么**：用户把哪些词/句改成了另一种说法？这说明用户的表达偏好。
4. **结构调整**：段落顺序有没有变？开头和结尾有没有重写？

## 输出格式

### 删除
- {具体描述：删了什么，为什么可能觉得多余}

### 新增
- {具体描述：加了什么，为什么可能觉得缺失}

### 替换
- "{原文}" → "{修改后}"（推测原因：{原因}）

### 结构调整
- {具体描述}
```

### Step 3: 跨篇提取共性规律

将所有文章的对比结果汇总，提取出现 ≥ 2 次的共性：

```
以下是 {N} 篇文章的初稿→终稿对比结果。

{Step 2 的所有对比结果}

请找出跨篇的共性修改规律。要求：
1. 只提取出现 2 次及以上的共性，个别文章的特殊修改不算
2. 规律要具体可执行，不要笼统的"写得更好"
3. 按以下三个维度分类

## 表达规律
用户在用词、句式、语气上的固定偏好。
示例："总是把四字成语改成口语""删掉所有'在这个XX的时代'开头"

## 结构规律
用户在文章结构上的固定模式。
示例："开头必须有真实经历""每段结尾加一句个人判断"

## 内容规律
用户总是加入或删除的内容类型。
示例："总是加入自己的真实数据""删掉泛泛的总结段落"
```

### Step 4: 写入迭代规律文件

```bash
RULES_DIR="$PROJECT/原稿存档/_迭代规律"
mkdir -p "$RULES_DIR"
PLATFORM="公众号"  # 根据实际平台
```

**如果规律文件已存在**，先读取旧规律：
```bash
cat "$RULES_DIR/$PLATFORM-迭代规律.md"
```

将新提取的规律与旧规律合并：
- 重复的规律保留一条
- 新发现的规律追加
- 旧规律中与新规律矛盾的，以新规律为准（用户的风格会演变）
- 更新 `last_updated` 和 `analyzed_count`

**写入文件**：

```bash
cat > "$RULES_DIR/$PLATFORM-迭代规律.md" << 'EOF'
---
type: 迭代规律
platform: {平台}
last_updated: {YYYY-MM-DD}
analyzed_count: {累计分析篇数}
---

# {平台}迭代规律

> 基于 {N} 篇文章的初稿→终稿对比，最后更新于 {日期}

## 表达规律
- {规律1}
- {规律2}
...

## 结构规律
- {规律1}
...

## 内容规律
- {规律1}
...

## 分析来源
| 文章 | 分析日期 |
|------|---------|
| [[文章名]] | YYYY-MM-DD |
EOF
```

### Step 5: 标记已分析

将本次分析过的原稿文件 frontmatter 中 `iterated` 改为 `true`：

```bash
# 对每个已分析的原稿文件
# 将 iterated: false 改为 iterated: true
sed -i '' 's/iterated: false/iterated: true/' "$DRAFT_DIR/{原稿文件名}.md"
```

### Step 6: 输出报告

向用户展示本次迭代学习的结果：

```
=== 迭代学习完成 ===

分析篇数：{N} 篇
涉及平台：{平台列表}

## 本次提取的规律

### 表达规律
- {列出}

### 结构规律
- {列出}

### 内容规律
- {列出}

规律已写入：项目/{项目名}/原稿存档/_迭代规律/{平台}-迭代规律.md
下次使用 content-generator 生成内容时会自动读取并遵守。
===========================
```

## 快捷用法

- "执行迭代学习" → 全平台扫描
- "学习我公众号的修改" → 只扫描公众号
- "进化一下" → 全平台扫描
- "看看迭代规律" → 读取并展示当前规律文件内容
