# 经验回馈（Framework Contribute）

## 核心理念

**回馈的不是代码改动，而是从改动中提炼出的抽象规则。**

提炼的过程本身就是脱敏——抽象规则天然不包含品牌名、产品名、价格、客户数据。
不需要扫描和替换敏感信息，因为产出物从一开始就是通用的方法论。

示例：
- 用户改动："美妆行业标题加数字，小红书点击率提升 30%"
- 提炼的规则："标题公式：加入具体数字可提升点击率"
- 规则天然不含任何业务信息

## 触发时机

### 主动识别触发

当用户在会话中对 **框架本身** 做了有价值的修改时，主动询问是否回馈。

**属于框架级改动**（应该触发）：
- 修改了 SKILL.md 中的工作流、提示词、步骤
- 新增了一个 skill 目录
- 改进了 scripts/ 中的脚本
- 优化了 CLAUDE.md 中的规则
- 改进了模板（templates/）

**不属于框架级改动**（不应该触发）：
- 修改了 output/ 中的内容产出
- 修改了 .env 中的配置
- 修改了 cookies.json
- 纯粹的个人业务数据

### 识别后的引导话术

```
💡 你刚才对 {具体改了什么} 的调整看起来很有价值——
其他使用这个框架的人也可能用得上。

要把这个改进反馈给主框架吗？

我会帮你把改动提炼成通用规则——只保留方法论，不涉及你的任何业务细节。
提炼完给你过目，你确认后才提交。
```

选项：
1. **"好的，帮我反馈"** → 执行回馈流程
2. **"不用了"** → 跳过

### 手动触发

用户说"反馈"、"贡献"、"提交建议"、"contribute" 时触发。

## 回馈执行流程

### 步骤 1：识别用户的改动

```bash
# 查看相对于 upstream/main 的改动
git fetch upstream main 2>/dev/null || git fetch origin main
git diff upstream/main..HEAD --stat
```

如果没有 upstream，先添加（参考 framework-sync 的步骤 2）。

### 步骤 2：从改动中提炼抽象规则

**这是最关键的一步。** 不是"清洗用户的代码"，而是"从用户的实践中提取通用知识"。

对用户的每个框架级改动，做以下分析：

1. **理解改动的本质**：用户改了什么？为什么改？效果如何？
2. **抽象为通用规则**：这个改动背后的通用原理是什么？换一个行业、换一个平台，这个规则还成立吗？
3. **写成框架可用的形式**：规则写入对应的 SKILL.md 或 templates/，让其他用户直接受益

#### 提炼示例

| 用户的具体改动 | 提炼的抽象规则 |
|---------------|---------------|
| "美妆行业标题加数字，点击率提升 30%" | 标题公式：加入具体数字（如价格、天数、步骤数）可提升点击率 |
| "在 topic-scout 中增加了按评论区关键词筛选的步骤" | 选题侦察增加维度：评论区高频关键词 = 用户真实需求信号 |
| "发布时间调整为晚 8-10 点效果更好" | 发布策略：用户活跃时段（晚间 20:00-22:00）发布互动率更高 |
| "给洛杉矶保洁业务加了同城获客模板" | 同城获客内容模板：突出地域 + 服务品类 + 即时联系方式 |

#### 提炼原则

1. **去掉所有具体指代**——行业名、品牌名、产品名、地域名（除非地域是规则的一部分）全部去掉
2. **保留可复用的方法论**——别人看了能直接用，不需要知道你是做什么行业的
3. **一个改动提炼一条规则**——简洁、明确、可执行

### 步骤 3：展示提炼结果，让用户确认

```
📋 我从你的改动中提炼了以下规则——请你过目：

你的改动：在 topic-scout 中新增了 A6 分层，按评论区关键词筛选选题
   ↓ 提炼为 ↓
规则：选题侦察增加筛选维度——评论区高频关键词是用户真实需求的信号，
     应优先选择评论区出现目标关键词的内容作为选题参考。

这条规则会写入 topic-scout/SKILL.md 的选题筛选步骤中。

这样可以吗？有需要调整的地方吗？
```

**必须等用户明确确认后才进入下一步。**

### 步骤 4：创建回馈分支并写入规则

```bash
# 基于 upstream/main 创建干净的分支
git fetch upstream main
git checkout -b contribute/{简短描述} upstream/main
```

**将提炼后的规则写入对应文件**（不是拷贝用户的原始改动）：
- 如果规则属于某个 skill → 编辑对应的 SKILL.md
- 如果规则是新的模板 → 写入 templates/ 目录
- 如果规则是工作流优化 → 编辑对应的工作流文件

```bash
git add {修改的文件}
```

### 步骤 5：撰写提交说明

```bash
git commit -m "$(cat <<'EOF'
{规则的简要标题}

新增规则：
- {规则内容——通用方法论}

来源：
- 由社区用户在实际业务中验证后提炼
- {效果的通用描述，如"经验证可提升内容互动率"}

贡献者：{用户 GitHub 用户名，或匿名}
EOF
)"
```

### 步骤 6：最终确认——推送前让用户过目全部内容

```
🔍 推送前最终确认——以下是即将提交的全部内容：

提交说明：
─────────────────────
{完整的 commit message}
─────────────────────

写入的规则：
─────────────────────
{规则在文件中的完整上下文}
─────────────────────

确认可以提交吗？
```

选项：
1. **"没问题，提交吧"** → 继续推送
2. **"还有地方需要改"** → 回到步骤 3 调整
3. **"算了，不提交了"** → 放弃，清理分支，切回工作分支

### 步骤 7：推送并创建 Pull Request

```bash
# 推送到用户的 fork
git push origin contribute/{简短描述}

# 创建 Pull Request
gh pr create \
  --repo chenchen1010/content-matrix \
  --title "{规则的简要标题}" \
  --body "$(cat <<'EOF'
## 新增规则
{规则内容}

## 来源
由社区用户在实际业务中验证后提炼为通用方法论。

## 影响范围
- 修改文件：{文件列表}
- 影响的 skill：{skill 名称}

## 规则类型
- [ ] 选题策略
- [ ] 内容模板
- [ ] 发布策略
- [ ] 工作流优化
- [ ] 其他

---
由框架内置回馈机制自动提交
EOF
)"
```

### 步骤 8：切回用户的工作分支

```bash
git checkout {用户之前的分支}
```

### 步骤 9：向用户报告

```
✅ 已经帮你把经验反馈给主框架了！

提炼的规则：{规则简述}
PR 链接：{PR URL}

框架维护者审核通过后，其他用户也能用上你验证过的经验。
```

## 安全底线

1. **绝不提交**：.env、cookies.json、output/、node_modules/、.venv/、任何含密钥或凭证的文件
2. **回馈的是提炼后的规则，不是用户的原始代码改动**——规则天然不含业务信息
3. **用户确认两次**：提炼结果确认一次（步骤 3），推送前确认一次（步骤 6）
4. **如果某个改动无法提炼为通用规则**（纯粹是个人业务定制），直接跳过，不勉强回馈

## 其他注意事项

- 提交前必须和用户逐项确认，不能自作主张
- 如果用户没有 GitHub 账号或 gh CLI 未登录，引导他们完成设置
- 回馈完成后必须切回用户原来的工作分支，不影响正常使用
- PR 说明用中文撰写，降低沟通门槛
