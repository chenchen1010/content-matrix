# 首次使用引导

## 何时触发

当检测到以下任一条件时，自动运行此 skill：

1. 项目目录下不存在 `~/.content-matrix/config.json`
2. 用户说「我是新来的」「怎么开始」「这是什么」「帮我看看怎么用」
3. 会话中首次交互且用户意图不明确

## 产品定位（对用户怎么说）

保持口语化，不要念文档。可参考以下说法：

```
这是一个小红书选题调研助手。

你告诉我你做什么业务，再给我 3 个你业务相关的关键词，
我会直接去小红书帮你做一版深度调研：
看这个行业现在在发什么、谁在发、哪些打法真的有效、用户在关心什么。

最后我会把结果整理成一份飞书云文档给你，
里面有行业分析、用户需求、选题建议，
还会直接附上 3 条示范性的小红书笔记。

你拿着这版内容先去发，拿到反馈后再继续迭代。

先简单介绍下你的业务，再给我 3 个种子关键词？
```

## Step 1: 收集输入

至少拿到以下信息：

1. 业务介绍（做什么、在哪个城市/区域、目标客户是谁）
2. 业务身份（官方/平台、商业机构、个人从业者、信息聚合平台、纯内容号）
3. 3 个种子关键词

补充规则：
- 如果用户只给了行业和城市没给关键词，主动帮他拟 3 个并确认
- 如果用户只给了 1-2 个词，继续追问补齐到 3 个
- 如果用户没有明确业务身份，要补问，因为这会影响后续对标对象和有效打法判断

## Step 2: 检测依赖并准备环境

本流程需要两个依赖：小红书搜索 + 飞书交付。

### 2a. 小红书 MCP

```bash
mcporter list 2>/dev/null | grep -i xiaohongshu
```

- 已启动 → 继续
- 未启动 / offline → 运行 `platform-login` skill 引导登录并启动 MCP
- 不要把终端命令整段复制给用户

对用户只说：
```
先帮你连上小红书，稍等一下。
```

### 2b. 飞书 lark-cli

```bash
command -v lark-cli &>/dev/null && echo "INSTALLED" || echo "NOT_INSTALLED"
```

- 未安装 → 安装并登录
- 已安装未登录 → 执行登录
- 已安装已登录 → 继续

安装/登录命令：
```bash
npm install -g @larksuite/cli
npx skills add larksuite/cli -y -g
lark-cli config init --new
lark-cli auth login --as user --scope "wiki:wiki wiki:node docx:document drive:drive search:docs_data:read bitable:app bitable:app:readonly im:message:send_as_bot"
```

对用户只说：
```
帮你连一下飞书，会弹浏览器让你授权一下。
```

## Step 3: 运行深度版小红书调研

调用 `topic-scout` skill，直接使用深度版，不做体验版降级。

执行目标：
- 平台固定：小红书
- 初始关键词：用户提供的 3 个种子关键词（全部要搜）
- 默认跑完整 4 轮迭代，除非提前达到重叠率停止条件
- 输出重点不是“爆款列表”，而是“有效打法 + 用户需求 + 选题机会”

本轮调研必须回答以下问题：
1. 这个行业在小红书上的内容现状是什么
2. 哪些玩家在发：官方号 / 蓝 V / 广告 / 素人号 / KOC / 矩阵号
3. 哪些打法是有效的
4. 用户在不同阶段分别在看什么、搜什么、担心什么
5. 哪些选题方向值得优先做

### 有效打法识别原则（必须显式执行）

重点观察以下几类内容：
- 带广告标识的笔记
- 蓝 V 官方号 / 品牌号
- 素人号种草内容
- 粉丝量较大的合作号 / 小 KOC 号

判断逻辑不是“创意是否新”，而是“打法是否有效”：
- 点赞是否明显高于同类平均
- 评论是否活跃，是否出现真实咨询/互动
- 同一品牌或同类玩家是否持续重复使用同类结构
- 是否形成“官方号 + 素人号 + KOC”协同

只要有效，哪怕内容同质化，也应视为可参考打法。

## Step 4: 生成 3 条示范性小红书笔记

调研完成后，调用 `content-generator` skill，生成 3 条示范内容。

要求：
- 平台固定：小红书
- 数量固定：3 条
- 来源基于 Step 3 的调研结果和爆款参考
- 必须覆盖不同选题方向，不要 3 条同质内容
- 优先覆盖不同用户阶段 / 不同打法类型，例如：
  - 一条偏研究/建立信任
  - 一条偏对比/选择
  - 一条偏顾虑/避坑
- 融入用户提供的业务信息，而不是只做空泛改写

这 3 条内容的定位是：
- 帮用户快速开始发布
- 验证选题方向是否能落成内容
- 作为第一轮测试样板，而不是最终定稿

## Step 5: 交付到飞书

最终交付物是“一份飞书云文档”，不是分散的多个文档。

### 5a. 创建飞书云文档

创建一份主文档，用于承接完整调研结果。

文档标题建议：
- `{城市}{行业}小红书选题调研`
- 或 `{业务名}小红书选题作战稿`

文档内应整合以下内容：
- 业务背景
- 行业现状
- 玩家格局（官方号 / 广告 / 蓝 V / 素人 / KOC）
- 有效打法分析
- 用户需求与搜索意图
- 选题建议
- 3 条示范笔记

### 5b. 创建多维表格并嵌入文档

创建一个多维表格，作为“选题库”。
表格为文档中的执行层，不作为单独最终交付物强调。

选题库至少承载：
- 选题
- 所属阶段
- 核心问题/痛点
- 推荐打法
- 推荐账号类型
- 参考案例
- 优先级

然后把这个多维表格嵌入主文档中。

### 5c. 把 3 条示范笔记写入文档

在主文档中追加一个“示范笔记”章节，直接放入 3 条生成内容，方便用户复制和开始发布。

### 5d. 给用户发飞书通知

发送一条飞书消息给用户，告诉他：
- 调研报告已经生成
- 文档里有选题分析和嵌入式选题库
- 附带 3 条示范笔记
- 建议先拿这 3 条去发，再根据反馈继续迭代

### 5e. 推荐的自动化落地方式

使用脚本 `content-matrix/scripts/deliver_xiaohongshu_report.py` 完成飞书交付。

```bash
python3 content-matrix/scripts/deliver_xiaohongshu_report.py content-matrix/scripts/sample_xiaohongshu_report_payload.json --notify
```

脚本职责：
- 创建“选题库”多维表格并写入选题 rows
- 创建飞书云文档并写入调研报告
- 在文档中放入选题库链接（飞书通常会自动预览）
- 在文档中写入 3 条示范笔记
- 可选发送一条飞书消息给用户自己

## Step 6: 收尾话术

可用以下风格回复用户：

```
搞定了，我已经把第一版调研结果整理到飞书里了。

里面你会看到：
- 小红书上的行业现状
- 同行和不同玩家的有效打法
- 用户在不同阶段关心什么
- 一份可直接用的选题库
- 3 条示范性小红书笔记

这一版不是最终答案，重点是先帮你把方向跑出来。
你可以先拿这几个选题和示范内容去发，看看反馈，
后面我们再根据真实数据继续迭代。
```

## 注意事项

- 默认直接跑深度版，不要再走 1 轮体验版
- 默认交付到飞书云文档，不强调 Obsidian
- 第一版重点是“选题”和“有效打法”，不是完整内容工厂
- 不要用“创意新不新”作为主要判断标准，要用“是否有效”作为判断标准
- 不要要求用户复制终端命令，全程由助手代执行
